《Effective Python》读书笔记（二）

发表于 2018-03-14 更新于 2019-12-17 分类于 tech

本文是《Effective Python 编写高质量Python代码的59个有效方法》一书的笔记的第二篇，主要涉及到的都是python的面向对象设计中的代码风格问题。

0x3. Classes and Inheritance

0x31. Prefer helper classes over bookkeeping with dictionaries and tuples

套用原书中的话：用来保存状态的数据结构一旦变得过于复杂，就应该将其拆解为类，以便提供更为明确的接口，并更好地封装数据。这样也能在接口与具体实现之间创建抽象层。

用tuple的坏处在于它是不可变的，而且通过下标索引实际上并不直观，对源码的可读性产生了较大的影响。而字典则是，当引入嵌套字典时，多次的引用也会降低代码的友好程度，毕竟python最大的特点就是极强的可读性吧。一个代替的方案是collections.namedtuple，可以用来表示不可变的简单结构，颇像不可变的js对象，具名元组的访问方式是通过成员引用而不是字典的那种key的方式，相比来说就十分直观了。以刚刚过去的N1CTF中，计算点到直线的距离为例，单独定义点和直线的类显得过于繁琐，为了方便使用对象的属性，可以用namedtuple：

具名元组的缺点在于使用不够灵活，不能提供默认值，只适合在简单类中使用。再复杂的结构，就得考虑用类来作封装了。

0x34. Initialize parent classes with super

在单继承中，python可以通过直接调用基类的__init__方法来实现从基类的构造：

但对于多继承的对象来说，这种写法就可能造成错误，最大的问题在于菱形继承时，共同的基础超类的构造方法会被重复调用。这里涉及到MRO(method resolution order)的标准，目的就是解决多继承中的构造顺序问题。

MRO属于python对象模型的内容，这里暂时不深入，在应用层代码的层面，需要了解的是super函数。引用标准文档中对于super函数的解释是：

Return a proxy object that delegates method calls to a parent or sibling class of type.

super返回的可能是类的超类，也可能是邻类，对象模型层面上，它返回的是类的MRO链上的下一个元素。MRO链上，第一个元素是当前类本身，最后一个元素是object类，MRO链通过一种拓扑排序将继承关系序列化，这样就能避免重复调用基类的构造函数，而具体的序列化方法根据版本的不同也有差异。在python中，初始化超类都应该用super函数。

这样，所有类的构造函数都会被调用且只调用一次。上述写法是python2的，而在3中，super函数可以用更简便的写法，避免写两次类名：

0x35. Use multiple inheritance only for mix-in utility classes

之前跟别人说过，Java相比其他语言要简单，说的就是它没有多继承的机制。关于多继承是否是面向对象的必要特性的争论已经不是一天两天了，我相信，作为将面向对象发扬光大的语言，Java的“单继承，多接口”的继承模式应该是最成熟的解决方案了。这里也可以看到，python尽管支持多继承，但推荐的mix-in风格仍然只是“单继承、多接口”的一种表现而已。

mix-in类是一种小型的类，很多时候甚至没有自己的实例属性和构造函数。得益于动态类型的运行时检查机制，可以方便地写出比较通用的工具函数，使得继承该mix-in类的子类也被赋予相应的行为。把简单的行为封装到mix-in组件里，就能用多个mix-in组合出复杂的行为。

0x4. Metaclasses and Attributes

学到一个新的名词：rule of least surprise，大意是应该尽量避免产生古怪行为的做法。元类(metaclass)大概是python最古怪的特性了，为了代码的可读性，也应该尽量避免太过特异地使用元类。

0x43. Use descriptors for reusable @property methods

python的描述符协议（descriptor protocol）由两个magic methods定义：object.__get__和object.__set__，可以用来构建一些通用的对象。

根据官方文档，两个方法的签名分别是：

object.__get__(self, instance, owner)
object.__set__(self, instance, value)

这种定义涉及到object.__getattribute__的工作方式（见下节），只要一个类定义了这两种方法，那么这个类的对象就称为描述符。

0x44. Use __getattr__, __getattribute__, __setattr__ for lazy attributes

这里先辨析一下类的一对magic methods的区别：__get__表示当本对象被引用时应该返回什么值，而__getattr__则是表示对象的属性被引用时的返回值，对于set那一对也是如此。因此，__get__用于定义描述符类型的对象，而__getattr__用于按需获取实例属性。

__getattribute__与__getattr__不同的地方在于，访问一个实例属性时，前者会被无条件调用，而后者只有在引发AttributeError的时候才会被调用。因此在用法上，这两者可以有如下的不同：

  • __getattr__可以按需生成属性，使用内置的setattr将属性绑定到对象上。当访问非法属性名时，应该抛出AttributeError
  • __getattribute__可以用于对象范围内的通用属性检查。当使用内置的hasattr和getattr函数时，该方法都会被调用

__setattr__方法不需要考虑对象是否存在该属性，因此python中只有一个mutator，但却有两个accessor。

__getattr__只在将要引发AttributeError的情况下才会被调用，而__getattribute__和__setattr__在每一次相关操作时都会被调用。使用这三个方法时都要注意无限递归的情况，特别是后两个，因为它们会被无条件调用，从而在方法内部出现循环引用最终造成调用栈深度溢出。解决的方法是用super().__getattribute__(name)从对象的属性字典中直接取到属性的引用，对其他两个方法同理。
